# 各个协议熵编码差异

熵编码的差异正是各代视频编码效率提升的**核心来源之一**。下面按**技术演进脉络**来讲，
重点放在**熵编码/解码方式本身、复杂度、上下文建模能力**上，方便横向比较。

---

一句话总览
> 从 H.264 → HEVC → VVC / AV1 / AVS3，熵编码从“简单二值 → 高度上下文自适应
> → 超大上下文+多模型”，压缩效率提升明显，但解码复杂度和实现难度也显著增加。

---

各标准熵编码方式对比
| 标准         | 熵编码方法                         | 核心特点                    |
| ------------ | ---------------------------------- | --------------------------- |
| H.264        | **CAVLC / CABAC**                  | 引入上下文自适应算术编码    |
| VP9          | **基于布尔算术编码**               | 比 CABAC 简化，但上下文有限 |
| HEVC (H.265) | **改进 CABAC**                     | 更细粒度上下文，效率更高    |
| AV1          | **Multi-symbol Arithmetic Coding** | 多符号、超多上下文          |
| VVC (H.266)  | **高度增强 CABAC**                 | 上下文爆炸式增长            |
| AVS2         | **CABAC-like**                     | 接近 HEVC，但略简           |
| AVS3         | **新一代算术编码**                 | 与 VVC/AV1 同代             |

---

**H.264**
* **CAVLC**（Baseline）：
  * 无算术编码
  * 解码快，但压缩率低
* **CABAC**（Main/High）：
  * 二值算术编码
  * 上下文数量有限（≈400）
  * 解码串行性强

**第一次把“上下文建模”引入主流视频编码**

---

**VP9**
* 使用 **Boolean Arithmetic Coding**
* 特点：
  * 概率模型固定/半自适应
  * 不是真正的 CABAC
  * 硬件友好、并行性较好

**效率略低于 HEVC CABAC，但实现简单**

---

**HEVC / H.265**
* **全面使用 CABAC**
* 改进点：
  * 更多语法元素
  * 上下文数量大幅增加（~1000）
  * 更精细的概率更新

相比 H.264：
* 同质量 **节省约 10–15% 码率（仅靠熵编码）**
* 解码复杂度明显增加

---

**AV1**
* **多符号算术编码（非二值）**
* 关键差异：
  * 一次编码多个符号（不是 bit-by-bit）
  * 上下文数量极多（几千级别）
  * 支持更复杂概率模型

代价：
* 解码器复杂
* 软件实现难
* 但**压缩效率非常高**

---

**VVC / H.266**
* **CABAC 的“极限版本”**
* 特点：
  * 上下文数量爆炸（上万）
  * 针对每种语法元素精细建模
  * 强烈依赖历史统计

结果：
* 比 HEVC 再降 **~10% 以上码率**
* **熵解码成为性能瓶颈**

---

**AVS2**
* CABAC 风格算术编码
* 设计目标：
  * 介于 HEVC 与 H.264 之间
  * 偏向工程可实现性

**复杂度略低于 HEVC**

---

**AVS3**

* 新一代算术编码
* 特点：
  * 上下文设计与 VVC / AV1 同代
  * 支持 AI/超高分辨率场景
  * 面向国产硬件生态优化

**压缩效率 ≈ AV1 / VVC 水平**

---

**本质差异总结（重点）**

1. 编码单位
    * H.264 / HEVC / VVC：**二值化（bin）**
    * AV1：**多符号直接编码**
2. 上下文数量
    ```
    H.264 < VP9 < HEVC < AVS2 < AV1 ≈ AVS3 < VVC
    ```
3. 解码并行性
    * CABAC：串行性强（难并行）
    * VP9 / AV1：更利于并行
4. 实现复杂度
    * AV1 / VVC / AVS3 ≫ HEVC ≫ H.264

---

> **现代编码器的性能瓶颈，已经从“变换/预测”转移到了“熵解码”。**

# 熵编码的基础知识

---

> **熵 = 描述一个不确定结果，平均需要多少信息量**

或者更口语一点：

> **熵 = “猜这个东西有多难”**

熵编码做的唯一一件事：用尽量少的 bit，表示一串“出现概率不一样的符号”。

---

## 为什么需要“熵”这个概念？

因为我们关心的问题是：

> **“最少需要多少 bit，才能无歧义地描述一件事？”**

这不是工程拍脑袋能回答的，
**熵就是这个问题的理论下限。**

---

## 从“确定 vs 不确定”开始（关键直觉）

**完全确定的事情 → 熵 = 0**

例子：
* 太阳明天从东方升起
* `1 + 1 = 2`

不需要任何信息去“告诉别人”，因为大家都知道。

**不确定性为 0 → 熵为 0**

---

**完全不确定的事情 → 熵大**

例子：
* 抛一枚公平硬币
  ```
  正面 / 反面（50% / 50%）
  ```
你至少要 **1 bit**：
```
0 → 正面
1 → 反面
```

**不确定性大 → 熵大**

---

## 熵和“概率”的关系（非常重要）

### 直觉规则（记住这三条）

1. **事情越可能发生 → 信息量越小**
2. **事情越罕见 → 信息量越大**
3. **平均信息量 = 熵**

---

## 举一个强烈对比的例子

### 例 1：天气预报

情况 A：
```
晴天：99%
下雨：1%
```

情况 B：
```
晴天：50%
下雨：50%
```

你直觉会觉得哪种“更有信息量”？

明显是 **情况 B**

这正是熵在度量的东西。

---

## 单个事件的信息量

我们需要一个规则：

> **概率越小，信息量越大**

信息论里用的是：
```
信息量 = -log2(概率)
```

不用记公式，只看几个结果：

| 概率  | 信息量 |
| ----- | ------ |
| 1     | 0 bit  |
| 1/2   | 1 bit  |
| 1/4   | 2 bit  |
| 1/8   | 3 bit  |

每减少一半概率，就多 1 bit 信息。

---

## 什么是“熵”？（核心）

> **熵 = 所有可能结果的“平均信息量”**

**换成人话：**
* 不是“最坏情况”
* 不是“最好情况”
* 而是：**长期来看，每次平均要说多少 bit**

---

### 一个完整例子（不用公式）

#### 抛硬币（公平）

```
正面：50% → 1 bit
反面：50% → 1 bit
```

平均：

```
熵 = 1 bit
```

---

#### 一个不公平硬币

```
正面：90% → ~0.15 bit
反面：10% → ~3.32 bit
```

平均下来：

```
熵 ≈ 0.47 bit
```

👉 注意：
**你不能每次都只用 0.47 bit**，
但**长期平均**可以做到。

---

## 熵不是“编码”，而是“下限”

这是最容易混淆的点。

* 熵 ≠ 某种编码方法
* 熵 = 所有无损编码的理论极限

> **任何编码方式，平均码长 ≥ 熵**

这叫 **香农第一定理（无噪声编码定理）**

---

## 为什么 Huffman 接近熵，但达不到？

原因只有一个：

> **Huffman 的码长必须是整数 bit**

而熵允许：
```
0.3 bit
2.7 bit
```

所以 Huffman：
```
平均码长 ≥ 熵 + 一点浪费
```

---

## 为什么算术编码能逼近熵？

因为它：
* 不给每个符号单独分配 bit
* 而是给“整个序列”分配 bit
* bit 可以“分摊”到小数级别

**长期平均码长 → 熵**

---

## 一个和视频编码强相关的直觉

### 变换 + 量化后：

```
0 出现 80%
±1 出现 15%
其他 5%
```

这意味着：
* 熵很低
* 还有大量压缩空间

**熵编码就是在“榨干这些空间”**

---

## 几个常见误解（一定要澄清）

* **❌ 熵 = 混乱程度**：这是物理学里的比喻，不适合信息论初学。
* **❌ 熵越大越好**：不对，**熵越小，越容易压缩**
* **❌ 熵是某种算法**：不对，**熵是极限，不是方法**

---

## 用一句话把所有东西连起来

> **信息论告诉我们：
> 世界本身能被压缩到什么程度（熵）；
> 编码算法只是努力去接近这个极限。**
