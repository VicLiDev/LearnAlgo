# 算术编码（Arithmetic Coding）

---

> **算术编码不是“一个符号→一串 bit”，
> 而是“一整串符号 → 一个区间里的一个数”。**

bit 只是用来描述这个数落在哪。

---

## 为什么它比 Huffman 更好？

**Huffman 的限制**
* 码长只能是 **整数 bit**
* 真实信息量可能是：
  * 0.3 bit
  * 2.7 bit
    Huffman 给不了

**算术编码的优势**
* **可以用“非整数 bit”**
* 平均码长 → **逼近熵极限**

---

## 算术编码的核心直觉（非常重要）

### 从一个区间开始

```
[0.0 , 1.0)
```

假设符号概率：
| 符号 | 概率  |
| ---- | ----- |
| A    | 0.5   |
| B    | 0.3   |
| C    | 0.2   |

区间划分：
```
[0.0 , 0.5) → A
[0.5 , 0.8) → B
[0.8 , 1.0) → C
```

---

### 编码 = 不断“缩小区间”

编码字符串 `"AB"`：

**第 1 个符号：A**
  ```
  区间 → [0.0 , 0.5)
  ```
**第 2 个符号：B**
  * 在 **当前区间内** 再划分：
    ```
    A → [0.0 , 0.25)
    B → [0.25, 0.40)
    C → [0.40, 0.50)
    ```
    选 B：
    ```
    最终区间 = [0.25 , 0.40)
    ```

---

### 输出什么？

只要输出 **任何一个落在这个区间内的数**，例如：
```
0.3
```

这个数 **唯一代表整个符号序列**

---

## 解码为什么能“同步”？

解码器：
1. 拿到这个数（比如 0.3）
2. 看它落在哪个区间 → 解出符号
3. 用同样的概率表重新划分
4. 继续

**编码器和解码器只要：**
* 概率模型一致
* 符号顺序一致

就绝不会出错。

## 一个极简算术编码模型（用于 demo）

为了 demo 简单，我们做这些限制：
* 固定符号集：`A B C`
* 固定概率（不自适应）
* 用 `double` 表示区间
* 不做 bit-level renormalization（教学版）

这不是工业级，只是**帮助理解**
