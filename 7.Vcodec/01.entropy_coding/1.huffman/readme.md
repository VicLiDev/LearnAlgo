# 霍夫曼（Huffman）编码/解码

---

> 霍夫曼编码 = 给高频符号分配短 bit 串，给低频符号分配长 bit 串，并且保证不歧义。

---

## 为什么不会“读错”？—— 前缀码

霍夫曼码有一个**硬性约束**：

> **任何一个码字，都不是另一个码字的前缀**

例如（合法）：
```
A = 0
B = 10
C = 110
D = 111
```

不合法的例子：
```
A = 0
B = 01   （A 是 B 的前缀）
```

有了前缀码，**解码可以从左到右无歧义地读 bit**

---

## 霍夫曼编码是怎么“算”出来的？

输入：符号 + 频率（或概率）

例子：
```
A : 5
B : 2
C : 1
D : 1
```

---

### 核心算法（贪心）

**不断合并“频率最小的两个节点”**

步骤：

**Step 1**
```
C(1), D(1) → 合并 → CD(2)
```

**Step 2**
```
B(2), CD(2) → 合并 → BCD(4)
```

**Step 3**
```
A(5), BCD(4) → 合并 → ROOT(9)
```

---

### 分配 bit（左 0，右 1）

（左右无所谓，只要编码解码一致）
```
          ROOT
         /    \
       A       BCD
              /   \
            B     CD
                  / \
                 C   D
```

得到编码：
```
A = 0
B = 10
C = 110
D = 111
```

---

## 编码 & 解码在干什么？

**编码**
```
输入符号串 → 查表 → 拼 bit
```
例如：
```
ABCD → 0 10 110 111
```

**解码**
```
bit 流 → 从根节点走树 → 遇到叶子输出符号
```
例如：
```
0110110111
↓
0   → A
10  → B
110 → C
111 → D
```

---

## 霍夫曼编码的优缺点

优点
* 简单
* 解码快
* 硬件/软件都容易实现
* 无专利风险

缺点（视频编码不用它的原因）
* 码长只能是 **整数 bit**
* 概率一变，整棵树要重建
* 压缩效率明显低于算术编码

现在主要用于：
* 教学
* PNG / DEFLATE
* 某些简单压缩
